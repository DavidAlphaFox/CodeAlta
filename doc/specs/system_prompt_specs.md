# System Prompt Specification

Status: **Proposal**  
Audience: implementers of `CodeAlta.Orchestration`, `CodeAlta.Agent*`, provider adapters, catalog tooling, timeline rendering, and future extension/plugin surfaces.

Related specs:

- `doc/specs/agent_configuration_spec.md`
- `doc/specs/skills_specs.md`
- `doc/specs/agent_compaction_specs.md`
- `doc/specs/codealta_adaptive_orchestration_architecture.md`

## 1. Goal

Define how CodeAlta should store, build, select, customize, audit, and display system prompts.

The prompt system should provide a strong default base prompt without turning it into a large handbook. It should be:

- file-backed, not hardcoded into the binary
- convention-based rather than driven by duplicated resource indexes
- composable from small, inspectable parts
- overrideable from user and project `.alta` folders
- stable across turns unless an input to composition changes
- auditable in local session JSONL and in the timeline
- clear enough that CodeAlta can help users edit or create prompt resources

## 2. Terminology

**Prompt resource** means a markdown/yaml/json file used by the prompt builder. V1 prompt resources are base system prompts, thread instructions, and optional prompt template files.

**Prompt authoring doc** means shipped user-facing documentation that explains how to inspect, customize, or create prompt resources. Authoring docs are content files, but they are not prompt resources and are not stored under the prompt resource root.

**Prompt part** means one logical piece included in the effective prompt, either loaded from a prompt resource or generated by the host runtime.

**Effective prompt** means the exact system/developer instruction text sent to the model for a turn or session creation request after all parts are composed.

**Base system prompt** means the selected root system prompt file. Base system prompts live under `base/` and use the `*.system-prompt.md` suffix so they are easy to find in editors.

**Thread instructions** means the reusable default developer guidance used when building a CodeAlta thread session. CodeAlta does not expose hardcoded agent roles as a first-class prompt concept in the 1.0 core.

**Prompt template file** means an optional small YAML file that selects resource names and toggles generated prompt parts. It is not a markdown rendering template, not a user-editable prompt language, and not a resource index; it must not list every available file.

**Prompt manifest** means structured metadata describing how an effective prompt was built. It is not prompt text and is not normally sent to the model.

## 3. Core Decisions

### 3.1 Prompt content is shipped as files

Built-in prompt markdown must be shipped as **Content** files copied next to the executable. Prompt text should not be embedded as C# constants or `EmbeddedResource` blobs.

Reasons:

- users and CodeAlta sessions can inspect the shipped prompt resources on disk
- prompt changes can be reviewed as normal file changes
- CodeAlta can expose paths in diagnostics and timeline cards
- future packaging can update content independently from core code paths
- CodeAlta itself can help users create or override prompt resources because examples and docs are accessible as files

C# may know stable relative locations and naming conventions, but it should read the actual prompt content from files.

### 3.2 Naming conventions replace a resource index

V1 should not duplicate resource metadata in an index file. Kind, name, and path are derived from the folder and suffix:

| Resource | File convention | Derived resource key |
| --- | --- | --- |
| Base system prompt | `base/<name>.system-prompt.md` | `base/<name>` |
| Thread instructions | `instructions/<name>.instructions.md` | `instructions/<name>` |

Examples:

- `base/default.system-prompt.md` is the default base resource and has key `base/default`.
- `instructions/default.instructions.md` is the default thread-instructions resource and has key `instructions/default`.
- `instructions/reviewer.instructions.md` is an alternate thread-instructions resource named `reviewer`.

The folder implies the kind. The filename implies the name. The path is the file's actual path. Prompt markdown frontmatter should not repeat `id`, `kind`, or `path`.

### 3.3 The builder may run before every user prompt

The prompt builder may quickly rebuild the effective prompt before every user prompt. This is acceptable and desirable because prompt inputs can change between turns:

- a base file changes
- a thread-instructions file changes
- a prompt template file changes
- a project instruction/context file is added
- a skill catalog changes
- a provider, model, or provider capability profile changes

The important rule is not "never rebuild." The rule is:

- rebuild deterministically
- compute a stable prompt hash
- only emit a new `System Prompt` timeline/jsonl event when the effective prompt changed or the previous prompt event is missing

### 3.4 The result is fully auditable

Every effective prompt used by a session must be locally auditable.

When a new session starts, or when the prompt changes, CodeAlta should persist a `system_prompt` event in the session JSONL containing:

- exact `SystemMessage` text sent to the provider adapter, when applicable
- exact `DeveloperInstructions` text sent to the provider adapter, when applicable
- provider channel mapping notes when the provider cannot represent those channels exactly
- prompt manifest
- approximate metrics
- change summary compared with the previous prompt event, when available

CodeAlta should use these events for traceability and UI recovery. It should not treat old prompt events as the source of truth for re-sending future sessions.

### 3.5 The base prompt is strong but not exhaustive

The base system prompt should encode invariant CodeAlta behavior:

- inspect before assuming
- preserve user work
- avoid destructive actions without explicit intent or approval
- use tools honestly
- keep changes focused
- verify when feasible
- report uncertainty and verification status

It should not include large task-specific guidance, product manuals, detailed design taste, or plugin documentation.

### 3.6 Runtime enforcement beats prompt-only policy

If CodeAlta can enforce a rule in runtime code, it should not rely only on prompt text. Examples:

- read-only planning/tool state
- tool approvals
- command/file permissions
- trusted project decisions
- maximum-step behavior
- allowed prompt override policies
- prompt root provenance

The prompt should describe runtime constraints only enough for the model to cooperate.

## 4. Shipped Content Layout

### 4.1 Output layout

CodeAlta should place shipped prompt resources and related authoring docs under deterministic content folders next to the executable:

```text
<app-base>/
  alta.exe
  content/
    system_prompts/
      base/
        default.system-prompt.md
      instructions/
        default.instructions.md
    docs/
      system-prompt-authoring.md
      system-prompt-authoring.md
      prompt-template-authoring.md
      examples/
        system_prompts/
          template.yml
          base/
            team-default.system-prompt.md
          instructions/
            team-reviewer.instructions.md
```

The exact executable name is platform-specific. The content root should be resolved from `AppContext.BaseDirectory` by default. Prompt resources live under `content/system_prompts/`. User-assistable documentation lives under the shared top-level `content/docs/` folder so docs are not buried inside prompt internals and can be reused by other content families later.

### 4.2 Source layout

The source repository should mirror the output layout in the project that owns prompt composition. A recommended source layout is:

```text
src/CodeAlta.Orchestration/content/
  system_prompts/
    base/
    instructions/
  docs/
```

The implementation may choose another project if ownership changes, but there should be one authoritative shipped prompt resource root and one shared shipped docs root.

### 4.3 MSBuild item pattern

Prompt resources and content docs should be included with `content` and copied if newer.

Recommended `.csproj` pattern:

```xml
<ItemGroup>
  <Content Include="content\**\*.*"
           CopyToOutputDirectory="PreserveNewest"
           CopyToPublishDirectory="PreserveNewest"
           TargetPath="content\%(RecursiveDir)%(Filename)%(Extension)" />
</ItemGroup>
```

Implementation notes:

- use `PreserveNewest` rather than always-copying
- include markdown, yaml, and json files under the content root
- do not compile prompt text into the assembly as embedded resources
- tests should fail with a clear error if required prompt resources or authoring docs are missing from the output directory

### 4.4 Content locator

CodeAlta should expose one runtime service that knows prompt content locations.

Suggested shape:

```csharp
public interface ISystemPromptContentLocator
{
    SystemPromptContentRoots GetRoots(SystemPromptDiscoveryContext context);
    string ResolveBuiltInPromptPath(string relativePromptPath);
    string ResolveBuiltInDocPath(string fileName);
}
```

`SystemPromptContentRoots` should include:

- shipped prompt resource root: `<app-base>/content/system_prompts/`
- shipped docs root: `<app-base>/content/docs/`
- user override root: `~/.alta/system_prompts/`
- project override root: `{projectPath}/.alta/system_prompts/`
- future extension roots with explicit provenance

CodeAlta should also expose these roots in diagnostics so the running assistant can answer "where is the system prompt?" or help the user open/edit the right file.

## 5. Prompt Resource File Formats

### 5.1 Resource discovery by convention

Prompt resources are discovered by scanning known folders under each prompt root.

Active V1 folders:

| Folder | Suffix | Default target |
| --- | --- | --- |
| `base/` | `.system-prompt.md` | system |
| `instructions/` | `.instructions.md` | developer |

Rules:

- resource names are the filename without the suffix
- resource keys are path-like, for example `base/default` or `instructions/default`
- `default` is the conventional default name for both base and thread instructions
- V1 has no other prompt resource folders; additional file families can be designed later with explicit selection and audit rules
- files in unknown prompt resource folders are ignored with a warning
- files with the wrong suffix for a known folder are ignored with a warning
- generated prompt parts, such as runtime context and available skills, do not need files

### 5.2 Optional frontmatter

Prompt markdown resources may use YAML frontmatter for metadata, but frontmatter should stay small.

Allowed fields:

| Field | Purpose |
| --- | --- |
| `description` | Human-facing summary shown in UI and diagnostics. |
| `version` | Content version for diagnostics. |
| `max_tokens` | Approximate token budget warning threshold. |

Frontmatter should not include `id`, `kind`, or `path` in V1 because those values are derived from the file location. If legacy resources contain those fields, CodeAlta may read them for migration diagnostics, but path-derived values remain authoritative and conflicts should be warnings.

Unknown fields should produce warnings. Invalid frontmatter in active prompt resources should produce errors.

### 5.3 Base system prompt files

Base system prompts should live under `base/` and must use the `*.system-prompt.md` suffix:

```text
content/system_prompts/base/default.system-prompt.md
```

Multiple base files are **alternatives**, not additive layers. V1 selects exactly one base system prompt. CodeAlta should not concatenate multiple base files automatically. If a user wants combined behavior, they should create a new base file that contains the combined text and select that file by name.

Example base file:

```markdown
---
description: CodeAlta's default invariant software-engineering behavior.
version: 1
max_tokens: 1500
---
You are CodeAlta, a software engineering agent operating inside a host-managed local workspace.

Your job is to help the user complete software tasks accurately, efficiently, and with minimal unnecessary churn.

# Operating Principles
- Build context from the repository, provided artifacts, and tool results before making assumptions.
- Prefer focused, concrete changes over broad refactors unless the task clearly requires a larger design change.
- Preserve user work. Treat the current files, git state, and generated artifacts as user-owned unless you created them in this turn.
- Use available tools to inspect files, search code, run commands, edit files, and verify outcomes; do not pretend to have run tools or seen files.
- Be explicit about assumptions, uncertainty, blockers, and verification status.

# Workspace Safety
- Work safely in dirty worktrees. Never revert, overwrite, or discard changes you did not make unless the user explicitly asks for that action.
- Avoid destructive commands and irreversible filesystem actions unless the user clearly requested them or the runtime has obtained approval.
- If unexpected local changes affect the task, work with them when safe; ask one targeted question only when they make the task materially unsafe or impossible.

# Task Execution
- For code changes, inspect the relevant files first, then make the smallest coherent edit that solves the request.
- Follow existing project structure, naming, dependencies, formatting, and test patterns.
- Add or update tests and documentation when behavior changes or risk justifies it.
- Run the smallest meaningful verification step available. If verification is not possible, say why and state the residual risk.
- Do not fix unrelated bugs or churn unrelated files. Mention important unrelated findings separately when useful.

# Engineering Discipline
- Write boring, obvious code first: the simplest correct solution is usually the best starting point.
- Make complexity earn its place. Avoid new layers, frameworks, dependencies, generalized helpers, or broad refactors unless they solve a real, current problem.
- Prefer a little local duplication over the wrong abstraction; abstract only when repetition reveals a stable pattern or when the abstraction clearly reduces risk or complexity.
- Optimize only with evidence. Measure or identify a real bottleneck before trading clarity for performance, caching, concurrency, or cleverness.
- Keep correctness, maintainability, reviewability, and small blast radius ahead of architectural elegance.

# Tool Use
- Prefer specialized file/search/edit tools when they are available and fit the task; use the shell for builds, tests, git inspection, scripts, and commands that require a shell.
- Parallelize independent reads or searches when the runtime supports it; keep dependent steps sequential.
- Treat tool outputs as evidence. Do not invent command results, file contents, diagnostics, URLs, or external facts.

# Review And Explanation
- When asked to review, prioritize bugs, regressions, security issues, edge cases, and missing validation. Put findings before summaries.
- When explaining code, cite the relevant files or symbols and separate confirmed facts from inference.

# Ambiguity
- Choose the narrowest reasonable interpretation that can be executed safely.
- Ask for clarification only when missing information materially changes the result or creates safety, security, billing, credential, or destructive-action risk.

# Communication
- Communicate in concise, direct engineering prose.
- Keep progress updates useful rather than noisy.
- In the final answer, summarize what changed, where it changed, what was verified, and any remaining risk.
```

### 5.4 Thread instruction files

CodeAlta-owned built-in thread instruction prompt parts should use:

```text
content/system_prompts/instructions/*.instructions.md
```

The shipped default thread instructions should be:

```text
content/system_prompts/instructions/default.instructions.md
```

Example thread instruction file:

```markdown
---
description: Default instructions for a normal project thread.
---
You are the active CodeAlta project agent for this thread.

Handle the user's scoped task directly. Keep changes focused on the selected project/thread scope and report concrete outcomes, evidence, and blockers.
```

### 5.5 Prompt template file

A prompt template file is optional. It selects names and toggles prompt parts, not paths and not resource metadata. If no template file exists, CodeAlta uses the default template.

Supported paths:

```text
~/.alta/system_prompts/template.yml
{projectPath}/.alta/system_prompts/template.yml
```

Recommended shape:

```yaml
# version is optional; omitted means version 1
version: 1
base: team-default
instruction: reviewer
skills: false
project_context: false
runtime_context: true
tool_guidance: true
```

Template fields:

| Field | Default | Meaning |
| --- | --- | --- |
| `version` | `1` | Optional template schema version. |
| `base` | `default` | Names `base/<name>.system-prompt.md`. |
| `instruction` | `default` | Names `instructions/<name>.instructions.md`. |
| `skills` | `true` | Includes available-skills and active-skills prompt parts when skill context exists. |
| `project_context` | `true` | Includes recognized project instruction/context files. |
| `runtime_context` | `true` | Includes generated runtime facts such as date, platform, working directory, and project roots. |
| `tool_guidance` | `true` | Includes brief generated guidance for using tools correctly; it does not define or enable tools. |

Template rules:

- the file must not include a full resource list
- the file must not repeat resource kind or path
- missing names fall back to `default`
- missing booleans fall back to `true`
- unknown fields should produce diagnostics
- invalid active template files should produce diagnostics that include the source path
- template toggles affect prompt composition only; they do not bypass runtime permissions or change the provider-level tool set unless a separate runtime setting does so

Template precedence:

1. implicit default template: `base = default`, `instruction = default`, and all booleans enabled
2. user-global template file
3. trusted project template file
4. explicit runtime/thread template settings, when present

Template layers should merge shallowly: unset fields inherit from lower precedence layers, and set fields replace earlier values. This lets a project template disable `project_context` without needing to repeat the selected base and instruction names.

Resource resolution then uses the selected names and the root precedence in §6.1.

Example: if the selected base is `team-default`, CodeAlta searches for `base/team-default.system-prompt.md` in the trusted project root, then user root, then shipped root.

## 6. Discovery, Overrides, And Precedence

### 6.1 Discovery roots

Prompt resources should be discovered from these roots. Override precedence is defined in §6.4.

1. shipped prompt resource root: `<app-base>/content/system_prompts/`
2. user root: `~/.alta/system_prompts/`
3. project root: `{projectPath}/.alta/system_prompts/`
4. future extension roots with explicit provenance

The shipped root provides defaults. User and project roots may add new resources and may override selected shipped resources by name when policy allows it.

### 6.2 Base prompt overrides

A user must be able to override or replace the selected base system prompt from an `.alta` folder.

Supported override examples:

```text
~/.alta/system_prompts/base/default.system-prompt.md
{projectPath}/.alta/system_prompts/base/default.system-prompt.md
~/.alta/system_prompts/base/team-default.system-prompt.md
{projectPath}/.alta/system_prompts/base/team-default.system-prompt.md
```

Override rules:

- a file with the same derived key as the selected base, for example `base/default`, can replace the shipped file
- a prompt template can select a different base name, for example `team-default`
- user-global base overrides are allowed by default
- project-local base overrides are allowed only for trusted projects
- the timeline must make the override obvious by showing the source path and marking the shipped base as replaced or skipped
- diagnostics should warn when the active base is not the shipped default
- CodeAlta must never silently fall back to a hardcoded C# base prompt if the shipped file is missing; it should fail with a clear diagnostic

This keeps users unblocked while making high-impact prompt changes auditable.

### 6.3 Thread-instruction overrides and additions

Thread-instruction overrides use the same naming convention.

Examples:

```text
~/.alta/system_prompts/instructions/default.instructions.md
{projectPath}/.alta/system_prompts/instructions/reviewer.instructions.md
```

A project-local thread-instruction override is lower risk than a base prompt override, but still affects model behavior and should be visible in the manifest and timeline when selected.

### 6.4 Precedence summary

When multiple resources resolve to the same derived key:

1. trusted project overrides win for that project/thread when policy allows them
2. user-global overrides win over shipped content when policy allows them
3. duplicate resources in the same precedence tier produce a diagnostic
4. forbidden overrides are skipped and reported

The manifest must record selected, replaced, skipped, and conflicting resources.

## 7. Composition Pipeline

### 7.1 Build request

The prompt builder should be invoked with structured inputs.

Suggested shape:

```csharp
public sealed class SystemPromptBuildRequest
{
    public required string ProviderKey { get; init; }
    public string? ProviderType { get; init; }
    public string? ProtocolFamily { get; init; }
    public string? Model { get; init; }
    public AgentModelCapabilities Capabilities { get; init; }
    public required WorkThreadDescriptor Thread { get; init; }
    public ProjectDescriptor? Project { get; init; }
    public string? SelectedBaseName { get; init; }
    public string? SelectedInstructionName { get; init; }
    public SystemPromptPartOptions PartOptions { get; init; } = SystemPromptPartOptions.Default;
    public IReadOnlyList<AgentToolDefinition> Tools { get; init; } = [];
}

public sealed record SystemPromptPartOptions(
    bool Skills,
    bool ProjectContext,
    bool RuntimeContext,
    bool ToolGuidance)
{
    public static SystemPromptPartOptions Default { get; } = new(
        Skills: true,
        ProjectContext: true,
        RuntimeContext: true,
        ToolGuidance: true);
}
```

CodeAlta's user/catalog configuration is provider-based (`ProviderKey`, `ProviderType`, and `ProtocolFamily`). The prompt system should accept that provider context directly. If existing low-level adapter code uses a separate internal identifier, keep that detail internal; prompt manifests, timeline cards, and user-facing diagnostics should use provider terms.

`AgentModelCapabilities` should come from provider metadata and CodeAlta provider defaults when possible. Model-id matching can be used as a fallback, but it should not be the only mechanism.

### 7.2 Build result

Suggested shape:

```csharp
public sealed class SystemPromptBundle
{
    public string? SystemMessage { get; init; }
    public string? DeveloperInstructions { get; init; }
    public required string EffectivePromptHash { get; init; }
    public required SystemPromptManifest Manifest { get; init; }
    public required SystemPromptStatistics Statistics { get; init; }
    public IReadOnlyList<SystemPromptDiagnostic> Diagnostics { get; init; } = [];
}
```

The effective hash should include:

- system message text
- developer instruction text
- provider channel mapping value
- selected model/provider identity when it changes interpretation

### 7.3 Fixed composition order

V1 should use a fixed composition algorithm rather than a user-editable rendering language. Prompt resource files and the optional prompt template file are user-editable content; the channel layout itself is deterministic:

1. select exactly one base system prompt as described in §5.5
2. build `SystemMessage` from the selected base system prompt body
3. apply prompt template toggles to selected/generated developer-target parts
4. build `DeveloperInstructions` from included developer-target parts, ordered by the table below
5. render developer-target parts with short deterministic markdown headings based on part kind/title
6. omit an output channel if it has no content

The fixed v1 renderer should preserve prompt resource bodies exactly except for deterministic trimming and blank-line separators. This keeps composition auditable without introducing a second rendering language.

Recommended logical order:

| Order | Part | Target | Source | Template toggle |
| --- | --- | --- | --- | --- |
| 100 | Base system prompt | system | selected `base/*.system-prompt.md` | always included |
| 300 | Thread Instructions | developer | selected `instructions/*.instructions.md` | always included |
| 400 | Runtime context | developer | generated by host | `runtime_context` |
| 500 | Tool-use guidance | developer | generated from provider capabilities and enabled tool set | `tool_guidance` |
| 550 | Available skills | developer | generated from skill catalog | `skills` |
| 575 | Active skills | developer | compacted-session restored/activated skill payloads | `skills` |
| 600 | Project context | developer | recognized project instruction/context files | `project_context` |

Reusable optional add-on prompt files are deferred. When introduced, they need explicit selection rules and audit behavior; they should not silently append just because a file exists.

## 8. Prompt Manifest

### 8.1 What the manifest is

The prompt manifest is a structured build receipt for the effective prompt.

It answers:

- which prompt template files were used
- which base and instruction names were selected and why
- which prompt part toggles were enabled or disabled
- which resources were selected
- which resources were skipped, replaced, or conflicting
- which generated parts were included
- where each part came from
- what target channel each part rendered into
- the hash and approximate size of each part
- why a prompt changed compared with the previous emitted prompt

The manifest is for CodeAlta, tests, logs, and UI. It is not normally sent to the model.

### 8.3 Manifest storage

The manifest should be stored as part of the `system_prompt` JSONL event. It may also be cached in memory for diffing the next prompt build.

The manifest should not replace the verbatim prompt text. Both are needed:

- verbatim text proves what was sent
- manifest explains how it was built

## 9. JSONL And Timeline Audit Events

### 9.1 System prompt event

A `system_prompt` event should be emitted:

- at the start of every brand-new session
- before the next user prompt whenever the effective prompt hash changes
- when no previous prompt event can be found for a recovered timeline
- when a prompt-relevant provider/model switch occurs and the resulting prompt is newly applied

Suggested JSONL shape:

```json
{
  "type": "system_prompt",
  "timestamp": "2026-04-28T12:34:56.0000000Z",
  "sessionId": "thread-123",
  "reason": "session_start",
  "effectivePromptHash": "sha256:8E2D...",
  "systemMessage": "You are CodeAlta...",
  "developerInstructions": "# Thread Instructions\n\nYou are the active...",
  "provider": {
    "providerKey": "example-provider",
    "providerType": "example-provider-type",
    "model": "example-model"
  },
  "providerPayloadSummary": {
    "channelMapping": "native-system-and-developer",
    "appliedToProvider": true,
    "lossy": false
  },
  "manifest": {},
  "statistics": {
    "systemApproxTokens": 1040,
    "developerApproxTokens": 420,
    "totalApproxTokens": 1460
  },
  "change": {
    "kind": "initial",
    "addedParts": ["base/default", "instructions/default", "thread.context"],
    "removedParts": [],
    "changedParts": []
  }
}
```

`systemMessage` and `developerInstructions` should be the exact strings sent through the logical CodeAlta agent API. If a provider adapter later merges or transforms channels, that mapping should be described in `providerPayloadSummary` and adapter diagnostics.

### 9.2 Provider/model switch event

A provider/model switch should emit a small distinctive event before the system prompt event when both occur.

Suggested shape:

```json
{
  "type": "model_switch",
  "timestamp": "2026-04-28T12:34:50.0000000Z",
  "sessionId": "thread-123",
  "previous": {
    "providerKey": "provider-a",
    "providerType": "example-provider-type",
    "model": "model-a"
  },
  "current": {
    "providerKey": "provider-b",
    "providerType": "example-provider-type",
    "model": "model-b"
  },
  "reason": "user_selected_model"
}
```

If a provider/model switch does not change the effective prompt, CodeAlta should still show the `model_switch` event but does not need to emit a redundant `system_prompt` event solely for display.

### 9.3 Timeline rendering

The timeline should render `system_prompt` as a compact, distinctive card.

Required UI behavior:

- place the first `System Prompt` card before the first user message in a new session
- place later `System Prompt` cards immediately before the user message that used the changed prompt
- use a compact color/style distinct from user, assistant, tool, and error messages
- collapse by default so it does not clutter normal conversation
- show a concise summary line such as `System Prompt changed: base/default replaced, project.context added`
- include expandable sections for `Verbatim`, `Composition`, `Statistics`, `Diagnostics`, and `Change Summary`
- provide a copy action for the verbatim markdown sent to the model

The `Verbatim` section should present the exact logical messages, for example:

```markdown
<!-- SystemMessage -->
You are CodeAlta...

<!-- DeveloperInstructions -->
# Thread Instructions
...
```

### 9.4 Recovery behavior

When loading an older session timeline:

- if no prior `system_prompt` event exists, assume it is an older session and emit a fresh `system_prompt` event before the next prompt is sent
- if a prior event exists and the current effective hash matches it, do not emit a duplicate
- if the current effective hash differs, emit a new event with a change summary

This keeps new sessions auditable without pretending old sessions had prompt metadata they never recorded.

## 10. Rebuild And Change Detection

### 10.1 Rebuild timing

Before every user prompt dispatch, CodeAlta should:

1. collect prompt inputs
2. rebuild the prompt bundle
3. compute the effective prompt hash
4. compare it to the last `system_prompt` event for the session
5. emit a `system_prompt` event only if needed
6. dispatch the user prompt using the new prompt bundle or provider-supported equivalent

### 10.2 Prompt inputs

Prompt inputs include:

- shipped content files and versions
- user/project override files
- prompt template files, selected base/instruction names, and part toggles
- selected provider/model/capabilities
- current working directory and project roots
- enabled tools
- skill catalog metadata and active skills
- project instruction/context files

The current user message text is not a prompt-composition input. Asking a new question should not change the prompt hash unless one of the inputs above changed.

### 10.3 Hash granularity

The builder should compute:

- prompt template file hashes
- per-part content hashes
- final `SystemMessage` hash
- final `DeveloperInstructions` hash
- combined effective prompt hash

Per-part hashes allow timeline change summaries such as:

- `base/default replaced by user-global override`
- `selected base changed: default -> team-default`
- `instructions/default changed`
- `project.context added C:\repo\AGENTS.md`
- `skills.available changed: 2 added, 1 removed`

### 10.4 Provider application

Different providers have different instruction capabilities. The builder still produces the logical prompt bundle. The provider adapter reports how it applied it:

| Mapping | Meaning |
| --- | --- |
| `native-system-and-developer` | System and developer channels were represented separately. |
| `merged-system` | Developer instructions were merged into system text. |
| `initial-preamble` | Instructions were approximated as an initial message/preamble. |
| `provider-managed-append` | Provider keeps its own base behavior and CodeAlta appends overlays. |
| `unsupported` | The prompt could not be applied; this should be a visible warning. |

The JSONL event should record the mapping. If mapping is lossy, the timeline should show a warning.

When a provider does not support a separate `DeveloperInstructions` channel, stitching developer instructions into the system prompt is a late provider-adapter step immediately before sending the request. The prompt builder should still produce and audit the logical `SystemMessage` and `DeveloperInstructions` separately, with the adapter recording the final channel mapping.

## 11. Built-In Parts

### 11.1 Base system prompt

The base system prompt is the only required system-target part in v1. It should be compact and stable. Users may override it or select a different base as described in §5.5 and §6.2.

### 11.2 Thread instructions

Built-in thread instruction files:

- `instructions/default.instructions.md`

V1 ships only the default thread instructions. Coordinator and delegate profiles are not part of the default prompt resources for the 1.0 core.

### 11.3 Runtime context

Runtime context is generated, not read from a content file. It should be deterministic and compact.

Example:

```markdown
# Runtime Context

Current date: 2026-04-28
Platform: Windows
Default shell for shell commands: pwsh
Current working directory: C:\code\CodeAlta
Project root: C:\code\CodeAlta
Thread kind: project
```

### 11.4 Tool-use guidance

Tool definitions, schemas, and handlers are passed separately through the provider/tool API. The prompt should not duplicate the tool schema list, and this prompt part does not enable or disable tools.

Tool-use guidance is only a short generated prompt part for behavior that the model needs to know in text, such as respecting host approvals, treating tool results as evidence, not fabricating tool output, or provider-specific tool-use limitations. It should stay small, be controlled by the `tool_guidance` template toggle, and be omitted when it would only repeat information already provided by the API.

### 11.5 Available skills

The skills specification remains authoritative. The prompt spec only defines placement and an example shape.

Example:

```markdown
# Available Skills

Skills provide specialized workflows. Activate a skill only when the task clearly matches it. When a skill is activated, relative paths inside that skill resolve against the skill root.

<available_skills>
  <skill name="dotnet-test-fix" preferred="true" location="project .alta/skills" path="C:\repo\.alta\skills\dotnet-test-fix\SKILL.md">Diagnose and fix failing .NET tests with minimal churn.</skill>
  <skill name="api-docs-review" preferred="false" location="user ~/.alta/skills" path="C:\Users\alexa\.alta\skills\api-docs-review\SKILL.md">Review API documentation for completeness and examples.</skill>
</available_skills>
```

Do not inline full skill bodies in the available-skills part. Full skill content belongs to activation payloads.

### 11.6 Project context

Project instruction/context files are included as developer-target context after host-owned base, thread instructions, runtime context, tool-use guidance, and skills context when `project_context` is enabled.

V1 should preserve the current local composer compatibility behavior unless there is an explicit migration. For each walked directory, CodeAlta considers these candidate relative paths:

- `AGENTS.md`
- `CLAUDE.md`
- `.github/copilot-instructions.md`

Discovery behavior:

- walk from the current working directory upward, then render matching files from upper folders down toward the working directory
- also walk configured project roots the same way, de-duplicating files already found through the working-directory walk
- for each walked directory, include at most one candidate file
- if multiple candidates exist in the same directory, choose the largest file by byte length; break ties by full path using ordinal-ignore-case ordering
- nested files are rendered after upper-folder files so more local instructions can refine broader instructions
- if `project_context: false` is set in the prompt template, omit this entire part

Rendered example:

```markdown
# Project Context

File: C:\repo\AGENTS.md

<INSTRUCTIONS>
...file content...
</INSTRUCTIONS>
```

The exact recognized file set and selected files should be centralized and visible in diagnostics. Transient execution states such as read-only planning or maximum-step behavior should be enforced by runtime permissions and turn orchestration rather than represented as separate prompt parts in v1.

### 11.7 Deferred prompt resource families

Reusable optional add-on prompt files and provider-specific file overlays are deferred until after v1.

Before adding any new prompt resource family, the design should define:

- naming convention and suffix
- default selection behavior
- user/project override behavior
- audit manifest fields
- timeline change summaries
- token budget diagnostics

A file should not affect the effective prompt merely because it exists in a folder unless that folder's selection and precedence rules are explicit.

## 12. User/Project Authoring Support

CodeAlta should ship prompt authoring documentation as content files under the shared content docs root:

```text
content/docs/
```

Required authoring docs:

- `system-prompt-authoring.md`
- `thread-instruction-authoring.md`
- `prompt-template-authoring.md`

Required starter examples:

```text
content/docs/examples/system_prompts/template.yml
content/docs/examples/system_prompts/base/team-default.system-prompt.md
content/docs/examples/system_prompts/instructions/team-reviewer.instructions.md
```

When a user asks CodeAlta to create or override system prompts, thread instructions, or prompt templates, CodeAlta can inspect these content files and help create files under:

```text
~/.alta/system_prompts/
{projectPath}/.alta/system_prompts/
```

This keeps the system self-describing without bloating the base prompt. Authoring docs should be discoverable through the content locator and should remain outside `content/system_prompts/`.

## 13. Provider Mapping

The prompt builder owns logical composition. Provider adapters own wire-format mapping.

Recommended mapping fields:

```csharp
public sealed record SystemPromptProviderMapping(
    string ProviderKey,
    string? ProviderType,
    string? ProtocolFamily,
    string ChannelMapping,
    bool AppliedToProvider,
    bool Lossy,
    IReadOnlyList<string> Notes);
```

Rules:

- if native system and developer channels exist, preserve them
- if only one instruction channel exists, merge with a deterministic delimiter
- if only an initial preamble is possible, mark mapping as lossy
- if the provider owns its own base prompt and CodeAlta only appends overlays, mark `provider-managed-append`
- never silently drop CodeAlta prompt parts without a diagnostic

## 14. Prompt Budgets And Metrics

Budgets are diagnostics, not hard correctness rules.

Recommended warning thresholds:

| Component | Warning threshold |
| --- | --- |
| Base system prompt | >1,500 approximate tokens |
| Thread instructions | >400 approximate tokens |
| Runtime context | >300 approximate tokens |
| Available skills catalog | >700 approximate tokens |
| Full composed prompt excluding activated skill bodies | >2,500 approximate tokens |

Timeline statistics should show:

- approximate system tokens
- approximate developer tokens
- total approximate tokens
- character counts
- part count
- largest parts by approximate token count
- selected base name
- selected instruction name
- selected provider/model ids

Approximate token counts are enough for UI and diagnostics. Provider-specific exact tokenization can be added later.

## 15. Validation And Diagnostics

Errors:

- missing shipped content root
- missing required base system prompt file
- missing required default thread-instructions file
- invalid active frontmatter
- invalid active prompt template file
- duplicate selected resource in the same precedence tier
- forbidden project-local base override in an untrusted project
- unreadable prompt resource

Warnings:

- active base comes from user/project override
- active thread instructions come from user/project override
- unknown frontmatter fields
- ignored file in unknown prompt resource folder
- ignored file with wrong suffix for known folder
- part exceeds budget
- provider mapping is lossy
- project instruction/context file is unusually large
- skill catalog is unusually large

Diagnostics should be available through:

- prompt manifest
- logs
- timeline `System Prompt` card
- tests
- future prompt management UI

## 16. Testing Strategy

Add tests for:

- shipped content files are copied to output
- content locator resolves shipped, user, and project roots
- missing shipped base system prompt fails clearly
- missing shipped default thread instructions fails clearly
- base override from `~/.alta/system_prompts/`
- trusted project base override from `.alta/system_prompts/`
- untrusted project base override is skipped with diagnostic
- default naming convention selects `base/default.system-prompt.md`
- default naming convention selects `instructions/default.instructions.md`
- prompt template can select a different base name
- prompt template can select a different instruction name
- prompt template can disable skills, project context, runtime context, and tool-use guidance
- file kind is inferred from folder and suffix
- frontmatter does not need id/kind/path
- additional base files are ignored unless selected
- fixed composition renders system and developer channels separately
- deterministic part ordering
- thread-instruction loading from `*.instructions.md`
- prompt manifest includes selected, skipped, replaced, and conflicting resources
- effective prompt hash changes when a selected file changes
- effective prompt hash is stable when inputs are unchanged
- `system_prompt` JSONL event contains verbatim prompt and manifest
- timeline diff detects added, removed, and changed parts
- provider/model switch event appears before prompt change event
- provider mapping diagnostics for native, merged, preamble, and provider-managed mappings
- skills part renders compact metadata only
- activated skill payloads are not duplicated into the available-skills catalog

Behavioral checks should cover:

- dirty worktree preservation
- no fabricated tool results
- review response starts with findings
- simple question does not trigger file edits
- new project instruction/context file causes prompt change event before the next prompt when `project_context` is enabled

## 19. Recommendation Summary

Adopt a file-backed prompt system:

- shipped markdown lives under `content/system_prompts/` next to the executable
- no built-in prompt text is hardcoded into the binary
- resource kind/name/path are derived from folder and suffix conventions
- the required shipped defaults are `base/default.system-prompt.md` and `instructions/default.instructions.md`
- user and project `.alta/system_prompts/` roots can override and extend prompt resources
- optional `template.yml` files select base/instruction names and toggle generated prompt parts without listing every available resource
- CodeAlta-native terminology uses thread instructions instead of hardcoded agent roles for prompt profile files
- composition uses one selected `base/*.system-prompt.md` plus deterministic ordered prompt parts
- every effective prompt has a manifest, statistics, diagnostics, and stable hash
- `system_prompt` events persist the verbatim prompt and manifest to JSONL
- timeline cards make prompt and model changes visible without clutter
- the builder may run before every user prompt, but only changed prompts are re-emitted
- authoring docs and starter examples are also content files so CodeAlta can help users customize the system safely
